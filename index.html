What are notifications?

They are aspects of the user interface that bring something to the notice of the user.

They are an important part of almost all digital products.

They help improve the user experience.

Types of notications

The first level of notification require immediate attention and user action.

The second level of notfications require attention from the user at specific times after some event.

The third level of notifications are designed to inform users of an event, but do not require urgent attention.

The fourth level of notifications are directly related to user interaction on an element.

And there are notifications that are just available or discoverable.

These are just a summary of some common types of notification.

You may have different names for them.

You may also categorise them slightly differently.

  • Lots of different types.
  • They have different user needs.
  • Some are more important than others.

Things to consider

Let's look at some things to consider when designing and building notifications.

  • What type of information is being communicated?
  • What is the urgency of the information?
  • Are user actions needed?
  • What actions will trigger it?
  • Where will it appear?
  • How will it appear?
  • Is it time-based or dismissable?

Accessibility considerations for designers

  • Does it use colour alone for meaning?
  • Does it have sufficient colour contrast?
  • Does it contain animation/flashing that can these be turned off by users?

Accessibility considerations for developers

  • Does it have an accessible name?
  • Does it need a role or live regions?
  • Does it need to be announced to assistive technologies?
  • If so, what is the appropriate time?
  • Does it need to be dismissable?
  • Does it need to recieve focus?
  • If so, where does focus go after it is dismissed?
  • Are keystrokes needed within it?
  • Can all actions be undertaken using keyboard only?
  • Does it need to trap some keystrokes?

In reality, some of the focus management considerations should be made by designers.

However, they are often overlooked.

Or it becomes the developer's job to make these important decisions.

Ideally, teams should collaborate to make key decisions.

Toast notifications

These are often used to inform users of some sort of successful action.

They often appear in the bottom of the viewport, animating up into view.

They're often time-based, so will only appear on screen for a short time period.

They normally do not require user action.

If user-intaction is needed, it should probably be a Push or Growl notification.

What about Toast notification accessibility?

All users need to be informed that their actions have been successful.

So, this information needs to be announced to screen reader users.

The announcement should be polite rather than assertive - just like for sighted users.

<div role="status">
  ...
</div>
<div aria-live="polite">
  ...
</div>

There should be no need to send focus to the notification.

This makes focus managing easy for this type of notification.

The notification should appear in close proximity to the user's action, for ZoomText users.

Push notifications

They are often used to inform users of something important.

For this reason, they often need attention and user action.

They could appear anywhere within the viewport.

There is always at least one user action associated with the notification.

They should not be time-based.

So, users must engage with the notification, even just to dismiss it.

What about Push notification accessibility?

Due to it's importance, it will need to receive user focus.

Focus needs to go the the notification container, the heading or the first action inside.

<div tabindex="-1">
  ...
</div>
<div>
  <h3 tabindex="-1">Notification heading</h3>
</div>
<div>
  <h3>Notification heading</h3>
  <button></button>
</div>

As it receives focus, it will need an accessible name.

<div aria-label="Something">
  ...
</div>
<div aria-labelledby="aaa">
  <h3 id="aaa">Notification heading</h3>
</div>
<div>
  <h3>Notification heading</h3>
  <button aria-label="Dismiss">X</button>
</div>

Also due to it's importance, it needs to be announced immediately.

<div role="alert">
  ...
</div>
<div aria-live="assertive">
  ...
</div>

The notification needs to be keyboard-only accessible.

If dismissed by the user, you need to decide the most intuitive place to send focus.

Modals

They demand some sort of user action - that is their entire purpose.

They generally appear over the top of the overall UI.

There is always at least one user action associated with the notification.

As with push notifications, users must engage with the notification, even just to dismiss it.

What about modal accessibility?

Almost all of the same things apply here as with push notifications.

There are some key decisions around where focus should be sent with the modal is triggered.

Should focus go to the modal container, the heading or the first focusable item.

But the biggest difference is keyboard activity within the modal.

As mentioned before, modals are designed to demand some sort of user-action.

This means we want them to remain focussed on the modal until at least one user action has taken place.

While the modal is open, keyboard users should never be allowed to interact with the page below.

This means that TAB and SHIFT + TAB should be "trapped" within the modal.

Unless we want to allow them access to the browser Chrome - address bar etc.

And like push notifications, you need to decide the most intuitive place to send focus.

Wrap up

Notifications should considered carefully - especially from accessibility perspective.

Teams need to consider exactly how users can and should engage with notifications.

And how and when these notifications are presented to different users.
